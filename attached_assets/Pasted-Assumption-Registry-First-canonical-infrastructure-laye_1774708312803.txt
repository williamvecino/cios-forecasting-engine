Assumption Registry

First canonical infrastructure layer

This is the foundation. Everything else depends on it.

What It Is

The Assumption Registry is a structured record of what the system is implicitly assuming when it generates a forecast or recommendation.

Not opinions.
Not signals.
Not outputs.

Assumptions are the conditions that must be true for the forecast to hold.

Why This Is Critical

Without this layer:

decisions cannot be audited
forecasts cannot be recalibrated
disagreements cannot be resolved
trust cannot scale

Every executive decision system requires this.

Where It Runs

Behind the scenes.

Not a workflow step.
Not a visible screen.

But the outputs must be accessible when needed.

When It Triggers

Automatically.

The system should create or update assumptions when:

a forecast is generated
a constraint is identified
a recommendation is produced
a scenario is run
Example — From Your Current Case

Decision:

Adoption likely constrained in first 6 months.

Underlying assumptions:

Payer coverage remains unresolved
Clinic staffing capacity remains limited
Secondary supplier validation is pending
Guideline update not yet issued

These assumptions must be recorded.

Behavioral Rules

Rule 1

Every forecast must generate at least one assumption.

Rule 2

Every primary constraint must create an assumption.

Rule 3

Every recommendation must depend on at least one assumption.

Rule 4

If an assumption changes status, the forecast must be recalculated.

Assumptions Behind This Decision

Why This Matters Operationally

Because when reality changes, the system must know which assumption broke.

Not which signal changed.

Example — Real Trigger

Event:

National coverage approved.

System response:

Find matching assumption.

Change status:

active → validated

Recalculate forecast.
