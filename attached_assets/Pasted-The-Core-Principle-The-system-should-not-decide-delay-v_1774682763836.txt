The Core Principle

The system should not decide:

delay vs launch

It should decide:

which outcome is most probable

That works for every domain.

The Universal Rule the System Must Follow

After computing probabilities, always do:

Select the highest-probability outcome
Explain why
Show the closest analog pattern
Show what would change the outcome

That is the judgment engine.

Not domain-specific.

The Minimal Architecture That Makes It Universal

You need four components.

1) Outcome Set

Every case defines possible outcomes.

Examples:

Launch timing:

Launch by 2026
Launch in 2027
Launch later

Adoption:

Low adoption
Moderate adoption
High adoption

Guideline:

Included
Deferred
Rejected
2) Probability Engine

Already exists.

Computes likelihood for each outcome.

3) Analog Matching

Finds similar historical cases.

Not mandatory on day one — can start simple.

4) Judgment Selector

This is the new piece.

It does:

Most likely outcome
Second most likely outcome
Margin between them
Confidence

The Decision Rule That Works Everywhere

Implement this:

IF max_probability >= 0.5

THEN

most_likely_outcome = outcome_with_highest_probability
confidence = High

ELSE

most_likely_outcome = outcome_with_highest_probability
confidence = Moderate
note = Competing outcomes remain plausible

No domain dependency.

What to Tell Replit

Use this instruction exactly.

We need to add a universal Judgment Layer that works for all case types. After computing probabilities for the defined outcomes, the system should automatically select the highest-probability outcome and generate an Executive Judgment block stating the most likely outcome, probability, confidence, key drivers, closest analog pattern, and reversal triggers. This layer must operate independently of domain or case type and should not assume timeline delays or any specific outcome. It should dynamically use the outcome set defined in each case.
