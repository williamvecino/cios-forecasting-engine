==================================================
WHAT TO BUILD NOW
==================================================

Build a new integrity layer:

Decision Derivation Enforcement Layer

Purpose:
Guarantee that every item on the Decide page is derived from the forecast, not generated independently or from templates.

==================================================
CORE RULE
==================================================

Every Decide-page item must follow this chain:

Signal
→ Event Gate
→ Forecast Constraint
→ Barrier / Action / Segment Output

If any output skips this chain, block it from rendering.

==================================================
REQUIRED DATA MODEL
==================================================

Add a DecisionItem model with these required fields:

- decision_id
- decision_type
- title
- rationale

- source_gate_id
- source_signal_id
- forecast_dependency

- severity_or_priority
- derived_from_forecast (true/false)

No decision item should exist without source_gate_id.

Suggested types:
- barrier
- action
- segment_assignment
- trigger_event
- monitoring_item

==================================================
DERIVATION RULES
==================================================

1. Every barrier must link to a gate
Example:
Operational Readiness = weak
→ Workflow barrier

2. Every action must link to a weak/moderate gate
Example:
Reimbursement Readiness = weak
→ Early payer engagement

3. Every segment assignment must link to gate profile
Example:
Strong infrastructure / high familiarity
→ Early adopter

4. Every trigger event must link to an unresolved gate
Example:
Coverage approval pending
→ Trigger event = policy change

==================================================
BLOCKING RULES
==================================================

Do not render a Decide-page item if:

- source_gate_id is missing
- source_signal_id is missing when required
- forecast_dependency is missing
- derived_from_forecast is false

==================================================
CONTRADICTION CHECKS
==================================================

Add a decision integrity validator before rendering the Decide page.

Validator rules:

1. If gate status = STRONG
Then matching barrier severity cannot be HIGH

2. If gate status = WEAK
Then at least one barrier or action must exist for that gate

3. If final forecast is constrained by a gate
Then Decide page must include at least one output tied to that gate

4. If Brand Outlook is strong but Final Forecast is low
Then Decide page should emphasize execution barriers, not evidence barriers

5. Positive evidence cannot render as "Evidence Barrier = High"

==================================================
SECTION-SPECIFIC DERIVATION RULES
==================================================

ADOPTION SEGMENTATION
Segment assignments must be based on gate profiles.

Examples:
- strong familiarity + strong access + strong workflow = early adopter
- moderate familiarity + moderate access = persuadable
- weak workflow + weak access = late mover
- not clinically relevant / no capability = resistant

BARRIER DIAGNOSIS
Barrier severity must come from gate weakness, not signal positivity alone.

READINESS TIMELINE
Timeline must come from:
- current gate strength
- unresolved triggers
- time-horizon feasibility gate

COMPETITIVE RISK
Must be tied to:
- competitor signal family
- gate-level vulnerability
Do not render generic "monitor competitor" text without forecast linkage.

GROWTH FEASIBILITY
Must be tied to:
- final forecast
- segment size
- access and operational gates
Not generic optimism/pessimism.

==================================================

In Step 3, under Final Forecast, add:

Forecast Meaning

Fields:
1. Interpretation

- Primary constraint = weakest major gate
- What would change the forecast = strongest unresolved gate or signal family

Example:
