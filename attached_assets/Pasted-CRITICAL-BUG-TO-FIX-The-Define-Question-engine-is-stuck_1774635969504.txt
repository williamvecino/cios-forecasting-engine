CRITICAL BUG TO FIX

The Define Question engine is stuck inheriting context from the currently active Mallia case.

Current problem:
When a new question is entered, the parser/interpreter is still reading from active case state, so subject/context values persist from the previous case. This causes context bleed and makes Step 1 non-independent.

This must be fixed at the state architecture level.

REQUIRED CORRECTION

Separate these into different state objects:

1. draftQuestionState
2. activeCaseState

RULES

1. Step 1 Define Question must always operate on draftQuestionState only.
2. draftQuestionState must initialize clean with no inherited subject, outcome, groups, therapy, disease, or case-linked context.
3. The parser must never fall back to activeCase values when extraction fails.
4. If a field is not extracted, leave it undefined and ask for clarification.
5. Only after the draft question is complete may the system:
   - create a new case from the draft question
   OR
   - explicitly replace/update the current case if the user confirms.

DO NOT auto-bind a new draft question to the existing active case before completion.

STATE REQUIREMENTS

Create or verify separate stores/slices for:
- draftQuestion
- activeCase
- forecastWorkspace

DRAFT QUESTION RESET REQUIREMENTS

Implement a true resetDraftQuestion() that clears:
- raw input
- parsed subject
- parsed outcome
- parsed groups
- parsed time horizon
- interpreted question
- missing fields
- completeness status
- question type
- any inherited therapy/disease metadata

Clear Question must call resetDraftQuestion(), not just clear the input text.

PARSER RULE

If parser cannot identify subject or context, return undefined.
Do not use activeCase.subject, activeCase.therapy, or any previous-case fallback.

HYDRATION RULE

If the route contains an existing case ID, the Define Question screen may display:
- currently active case summary
but the editable draft question must still be independent unless the user explicitly chooses “edit this case question”.

Add an explicit mode flag:
- mode = "new_draft"
- mode = "edit_existing_case"

Default must be:
mode = "new_draft"

CASE CREATION RULE

On Continue from Step 1:
- if mode = new_draft, create a new case from draftQuestion
- do not mutate the active case automatically
- only switch active case after new case creation succeeds
