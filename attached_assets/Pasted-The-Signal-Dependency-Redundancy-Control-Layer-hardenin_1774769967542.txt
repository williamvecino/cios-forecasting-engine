The Signal Dependency & Redundancy Control Layer hardening phase is now complete and accepted as the current canonical baseline.

## Decision

Proceed to the **next canonical build step: Forecast Ledger**.

Do **not** jump to segmentation, barrier diagnosis, readiness timeline, competitive risk, growth feasibility, or reaction simulation yet. The next build remains the Forecast Ledger because dependency control now protects inference integrity and the system is ready to start tracking prediction quality over time.

## Forecast Ledger purpose

Create the calibration and accountability layer for CIOS so every forecast becomes a tracked, reviewable, updateable decision record.

The goal is not just to store outputs. The goal is to let CIOS learn whether its forecasts were:

* accurate
* overconfident
* underconfident
* improved or worsened by updates
* fragile due to concentration or low diversity

## Build requirements

### 1) Create Forecast Ledger data structure

Each forecast entry must persist at minimum:

* `ForecastID`
* `CaseID`
* `QuestionText`
* `DecisionDomain`
* `ForecastDate`
* `ForecastHorizon`
* `PriorProbability`
* `PosteriorProbability`
* `ConfidenceLevel`
* `ConfidenceCeilingApplied`
* `EvidenceDiversityScore`
* `PosteriorFragilityScore`
* `ConcentrationPenalty`
* `IndependentEvidenceFamilyCount`
* `RawSignalCount`
* `CompressedSignalCount`
* `KeyDriversSummary`
* `TopLineageClusters`
* `CounterSignalsSummary`
* `EnvironmentAdjustments`
* `UpdateVersion`
* `UpdateRationale`
* `ResolutionStatus`
* `ResolutionDate`
* `ResolvedOutcome`
* `BrierScore`
* `CalibrationBucket`

### 2) Support versioned forecast updates

The ledger must preserve forecast history, not overwrite it.

For any case:

* first forecast = version 1
* each later revision = new version entry
* preserve timestamp and rationale for each update
* show movement from prior version to new version

I want a permanent history of:

* what changed
* why it changed
* which signals drove the change
* whether the update improved calibration

### 3) Add forecast resolution workflow

Build a simple resolution process where a forecast can later be marked:

* Resolved True
* Resolved False
* Partially Resolved
* Still Open
* Not Resolvable

Resolved outcomes should support later scoring and calibration analysis.

### 4) Add Brier scoring and basic calibration tracking

Once forecasts resolve, calculate:

* Brier score per forecast
* average Brier score by question type / domain if available
* basic calibration bucket assignment (for example 0–10%, 11–20%, etc.)

Do not overbuild analytics yet. Start with correct storage and correct scoring.

### 5) Add forecast snapshot capture

Every saved forecast must include the inference state at time of save:

* prior
* posterior
* confidence level
* confidence ceiling state
* diversity / fragility / concentration metrics
* major lineage clusters
* major positive and negative drivers

This is important. The ledger must preserve the reasoning state that produced the forecast.

### 6) Add basic Forecast Ledger UI

Need a simple, usable ledger view showing:

* forecast date
* question
* version
* posterior
* confidence
* status
* resolved outcome
* Brier score if resolved

And on click:

* full forecast detail
* update history
* key drivers
* dependency-control summary
* rationale log

Functional first, polished later.

### 7) Add update-rationale requirement

When a forecast is revised, require a short rationale field.

Examples:

* new payer access signal
* updated guideline interpretation
* new competitor event
* workflow resistance lower than expected
* field sentiment weakening

This is essential for learning, not just record-keeping.

## Important architectural rule

The Forecast Ledger is not a reporting add-on. It is the calibration memory of CIOS.

## Acceptance criteria

Return with:

1. Forecast Ledger schema
2. Save/load flow working
3. Versioned forecast history working
4. Resolution workflow working
5. Brier scoring working
6. Basic ledger UI working
7. Snapshot capture of dependency-control metrics included
