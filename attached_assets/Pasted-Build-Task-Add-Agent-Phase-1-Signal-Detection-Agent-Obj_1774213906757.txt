Build Task — Add Agent Phase 1: Signal Detection Agent

Objective

Add a bounded agent that scans specified sources, extracts candidate signals, classifies them, suggests likely affected cases, and presents everything for human review before any signal enters active forecasting.

This agent must NOT update posterior probability directly.
This agent must NOT insert signals into active cases automatically.
It should only produce structured candidate signals for review.

Core role

The agent acts as:
- source scanning layer
- candidate signal extraction layer
- signal classification layer
- case suggestion layer

It is not the forecasting engine.

Architecture placement

Source scan
→ Signal Detection Agent
→ CandidateSignal output
→ Human review / validation
→ CIOS active signal set

New component

SignalDetectionAgent

Inputs

- sourceList
- optional therapyArea
- optional geography
- optional targetType
- optional specialty
- optional subspecialty
- optional dateRange
- optional caseUniverse (existing open/active cases)

Outputs

- detectedSignals
- extractionSummary
- likelyAffectedCases
- reviewQueue

Supported source types for v1

- manually entered URLs
- uploaded text/doc content
- curated source list entries
- basic public page fetches if already supported in the app

Do not build unrestricted crawling in v1.

Behavior rules

1) Scan specified sources
For each source:
- capture source label
- capture source URL if applicable
- capture publication/observation date if available
- extract text or snippet candidates

2) Detect candidate signals
Find statements that may represent adoption-relevant events such as:
- clinical data release
- conference reaction
- regulatory event
- payer/access event
- competitor event
- institutional readiness event
- KOL/supportive commentary
- safety concern
- operational friction

3) Classify each candidate signal
Each detected signal should include:
- signalType
- suggestedDirection (positive | negative | neutral)
- suggestedStrength (low | medium | high)
- suggestedScope (market | specialty | subspecialty | institution | physician)
- extractionConfidence (low | medium | high)
- possibleEventFamily
- detectedDate
- evidenceSnippet
- sourceLabel
- sourceUrl

4) Suggest likely affected cases
Match candidate signals to current case universe using:
- therapy area overlap
- geography overlap
- specialty/subspecialty overlap
- target type overlap
- institution/physician overlap if available

Output:
- list of possibly affected case IDs with confidence level

5) Human validation gate
Every detected signal must remain:
status = "candidate"
until a user validates it.

6) No autonomous forecast actions
The agent must not:
- assign final LR values
- compute posterior updates
- activate signals in cases automatically
- modify recommendations automatically
