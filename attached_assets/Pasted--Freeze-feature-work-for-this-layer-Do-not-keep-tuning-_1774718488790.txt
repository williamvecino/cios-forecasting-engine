4) Define pass/fail rules

You need binary integrity rules.

Examples:

State sync rule

Step 3 must use the exact latest Step 2 saved state.

Confidence rule

If unresolved contradictions or major unresolved gates exist, confidence cannot be “high” unless there is explicit low-sensitivity proof.

Headline rule

If positive weighted support exceeds negative weighted drag and no hard gate exists, the headline cannot say “barriers outweigh supporting factors” unless the cap math explicitly justifies it.

Recommendation rule

A strong negative planning recommendation cannot appear unless final probability is below threshold and the barrier structure justifies low investment.

Narrative rule

Narrative statements must map to current structured state, not cached summaries.

These are machine-testable.

5) Run three test modes

You should not test only with normal usage.

A. Static regression tests

Same exact case every run.
Should produce same result every time.

This tests deterministic behavior.

B. Sensitivity tests

Change only one variable:

one signal strength
one gate
one confidence score
one time horizon

Then verify the output changes in the expected direction and magnitude.

This tests causal sanity.

C. Contradiction tests

Intentionally create cases like:

many positives but one unresolved gate
weak evidence but strong access
equal positives and negatives

Then check whether the system explains uncertainty correctly.

This tests judgment realism.

What the 6 canonical things help with

Yes, they will help, but in a specific order.

From the canonical build sequence you locked, the ones that help most immediately are:

1) Forecast Ledger

This is first because it gives you:

persistent case history
before/after comparisons
update tracking
visible drift detection

Without this, you cannot tell whether the system improved or just changed.

2) Calibration / Performance Dashboard

This is second because it lets you measure:

overconfidence
underconfidence
systematic pessimism
systematic gate over-penalization

Right now you are feeling that something is wrong.
This layer turns that into measurable evidence.

3) Adoption Segmentation Panel

This will help because some “low adoption” calls may actually be wrong framing.
The truth may be:

low broad adoption
but high early-adopter specialty uptake

That distinction matters a lot in ARIKAYCE-type cases.

4) Barrier Diagnosis Panel

This will force the system to separate:

evidence barrier
access barrier
workflow barrier
behavior barrier

That reduces double counting and vague “barriers outweigh support” language.

5) Readiness Timeline

This helps because a lot of your current disagreement is probably not about adoption per se, but about timing.
A system that only outputs one static probability compresses too much.

6) Competitive Risk / Growth Feasibility

These matter later, but they are not the immediate fix for your current contradiction problem.
