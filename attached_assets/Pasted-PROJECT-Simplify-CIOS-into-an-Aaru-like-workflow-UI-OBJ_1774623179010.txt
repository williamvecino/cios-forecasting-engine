PROJECT: Simplify CIOS into an Aaru-like workflow UI

OBJECTIVE
Restructure the UI so the product feels simple, guided, and decision-first.

The user should no longer see a collection of competing modules.
The user should experience one clear workflow:

1. Define Question
2. Add Information
3. See Forecast
4. Decide

All existing complexity should be reorganized underneath that flow.

CORE DESIGN RULE
One visible path. Hidden complexity.

The interface must feel like a guided decision system, not a toolbox.

PRIMARY UX GOAL
A new user should know within 3 seconds:
- where to start
- what this step is for
- what to do next

NEW INFORMATION ARCHITECTURE

TOP-LEVEL NAVIGATION
Replace or simplify current navigation to:

- Home
- Forecasts
- Library
- System

Do not expose many modules at top level.

PAGE PURPOSES

1. Home
Purpose:
Start or resume work

Contents:
- one main input box: “What do you want to forecast?”
- recent forecasts
- sample good questions
- resume last active forecast

Do not show:
- calibration tools
- orchestration tools
- agent tools
- scenario infrastructure
- technical dashboards

2. Forecasts
Purpose:
Access all forecast cases

Each forecast should open into the 4-step workflow:
- Define Question
- Add Information
- See Forecast
- Decide

3. Library
Purpose:
Access saved cases, example questions, templates, and historical reference items

Contents:
- saved forecast cases
- sample good questions
- templates by question type
- case library later if available

4. System
Purpose:
Hold advanced infrastructure and validation tools

Place here:
- Forecast Ledger
- Calibration / Performance Dashboard
- advanced scenario management
- actor simulation controls
- execution logs
- orchestration infrastructure
- dependency map
- system validation

MAIN PRODUCT FLOW

Every forecast should use this stepper:
1. Define Question
2. Add Information
3. See Forecast
4. Decide

Do not show Question and Case as separate starting objects anymore.
The user starts with a question.
Once structured, it becomes the case automatically.

STEP 1 — DEFINE QUESTION

Purpose:
Turn rough user intent into a forecast-ready question

UI REQUIREMENTS
Screen contents:
- large text input at top
- interpretation card below
- one clarification prompt at a time
- primary button: Continue
- button disabled until required fields are complete

TITLE
Define the question

SUBTITLE
Turn a rough idea into a forecast-ready decision question.

MAIN INPUT PLACEHOLDER
Example: Which regions will adopt first-line ARIKAYCE fastest in 12 months?

INTERPRETATION CARD FIELDS
- Question type
- Subject
- Outcome
- Groups being evaluated
- Time horizon
- Success definition
- Status

RIGHT-SIDE OR SECONDARY PANEL
Good questions
Examples:
- Will community pulmonologists adopt within 12 months?
- Will payer restrictions delay uptake by more than 6 months?
- Which stakeholder segment is most likely to adopt first?

Avoid
- Will this do well?
- What happens in the market?
- Will doctors like it?

BEHAVIOR
- user enters rough question
- system classifies question
- system extracts fields
- system identifies missing fields
- ask only one missing clarification at a time
- once complete, save as structured decision question
- auto-create case from it

IMPORTANT
Do not ask users to choose between “new case” and “ask a question.”
That distinction should disappear from the visible UX.

STEP 2 — ADD INFORMATION

Purpose:
Help the user understand what is pushing the forecast up or down

THIS STEP MUST BE DRIVER-FIRST
Do not start with a flat signal list.
Start with driver groups.

NEW SCREEN STRUCTURE
Three-column layout:

LEFT COLUMN
Driver groups

CENTER COLUMN
Signals/evidence attached to selected driver

RIGHT COLUMN
System summary of what currently matters most

LEFT COLUMN DRIVER GROUPS
Use intuitive buckets:
- Clinical Evidence
- Access and Reimbursement
- Stakeholder Behavior
- Competitive Pressure
- Operational Readiness

Inside each bucket:
- Primary Drivers
- Supporting Signals

Do not present raw technical taxonomies first.

CENTER COLUMN
When a driver is selected, show:
- signals attached to that driver
- direction
- strength
- reliability
- editable notes
- add signal button
- AI/system-suggested signals, but editable

RIGHT COLUMN
Show short interpretation:
- strongest upward pressure
- strongest downward pressure
- missing critical information
- current confidence level

PAGE TITLE
Add information

SUBTITLE
Add the main forces that could change the outcome.

IMPORTANT BEHAVIOR
User should always understand:
- which driver they are editing
- what evidence is attached
- whether it pushes up or down
- what is still missing

STEP 3 — SEE FORECAST

Purpose:
Show the current forecast clearly

The screen should answer only:
- What is the probability?
- Why?
- What could change it?
- Who matters most?

MAIN SCREEN LAYOUT

TOP SECTION
- forecast probability
- confidence level
- one-sentence interpretation

MIDDLE SECTION
- top 3 upward drivers
- top 3 downward drivers
- top affected segments or stakeholders

BOTTOM TABS
- Sensitivity
- Scenarios
- Segment View
- Calibration History

IMPORTANT
Scenarios should live here as a tab, not as a separate prominent product section.
Keep the workflow simple.

PAGE TITLE
See forecast

SUBTITLE
Review the current forecast and the forces shaping it.

STEP 4 — DECIDE

Purpose:
Translate forecast into action

Move all decision panels here.
Do not leave them floating as separate primary experiences.

PANELS TO PLACE UNDER DECIDE
- Adoption Segmentation Panel
- Barrier Diagnosis Panel
- Readiness Timeline
- Competitive Risk Panel
- Growth Feasibility Panel

OPTIONAL LATER CARD
- Test stakeholder reactions

This can later connect to actor simulation without cluttering the top navigation.

PAGE TITLE
Decide

SUBTITLE
See where to focus, what could block success, and what to watch next.

DECIDE SCREEN OUTPUT
- most important opportunity
- biggest risk
- likely barriers
- recommended focus areas
- what would materially change the forecast

- standalone calibration page in the main path
- standalone scenario workspace in the main path
- orchestration/system components

If these exist already, move them under:
System
or
advanced panels inside the workflow

QUESTION + CASE MERGE

Implement this product rule:
Question definition creates the case automatically.

USER EXPERIENCE
Before:
- Ask a question
- Create a case
- maybe do adopter discovery
- maybe add signals

After:
- Define Question
- Add Information
- See Forecast
- Decide

This is mandatory.

VISUAL / STRUCTURAL RULES

1. One primary action per screen
2. One sentence subtitle explaining the purpose of the screen
3. Stepper always visible at top of forecast workflow
4. Keep labels plain language
5. Hide technical complexity from first-time users
6. Do not show large numbers of cards/modules at once
7. At every screen, user must know what comes next

STATE / ROUTING RULES

1. Each forecast has one workflow state
2. Define Question is always first
3. Add Information cannot be accessed until the question is complete
4. See Forecast can render partially, but should indicate low confidence if information is sparse
5. Decide depends on forecast output
6. Preserve user progress in each step
7. Resume last active forecast from Home

COPY RULES

Use simple labels.
Examples:
- Define Question
- Add Information
- See Forecast
- Decide
- What do you want to forecast?
- What is pushing this outcome up or down?
- Review the current forecast
- Where should we focus?

Avoid jargon-heavy labels on the main workflow.

PROMINENCE RULE

Most prominent items:
- question input
- workflow stepper
- current forecast
- key drivers
- decision panels

Less prominent:
- ledger
- calibration
- orchestration
- execution logs
- advanced simulation infrastructure

SUCCESS CRITERIA

This redesign is successful if:
1. a new user can start from one obvious place
2. question and case no longer feel redundant
3. signal entry feels organized around drivers, not raw lists
4. forecast view is clear and focused
5. decision panels feel like the logical final step
6. advanced infrastructure is still accessible but no longer confusing the main path

IMPLEMENTATION ORDER

1. Simplify top-level navigation
2. Redesign Home
3. Merge Question + Case into one flow
4. Add 4-step workflow shell
5. Rebuild Add Information into driver-first layout
6. Reposition Forecast view
7. Move decision panels under Decide
8. Move advanced tools under System

IMPORTANT
Do not rebuild the entire app at once.
Wrap the current product in this structure first.
Then relocate components step by step.

FINAL DESIGN PRINCIPLE
The product should feel like:
a simple guided decision workflow

not:
a collection of forecasting tools
