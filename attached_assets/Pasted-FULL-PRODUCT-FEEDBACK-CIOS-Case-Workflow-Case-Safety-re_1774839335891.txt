FULL PRODUCT FEEDBACK — CIOS Case Workflow

Case: Safety-related label change risk for Xarelto within 12 months
Domain: Regulatory / Pharmacovigilance / Market Access
Objective: Improve decision-grade forecasting, simulation, and traceability

1) Simulation Workflow — Missing Core Scenario Definition
Problem

The system defines scenario parameters but not the scenario itself.

Current inputs:

Scenario Type
Message Source
Impact Level
Time Frame
Confidence Level

These are attributes.
They do not define the event being simulated.

This creates traceability and governance risk.

Required Additions — Simulation Scenario Section
Add Field — Scenario Name (required)

Place at the top of Step 1.

Field:

Scenario Name

Purpose:

Defines the event being simulated.

Examples:

FDA initiates formal safety review of rivaroxaban GI bleeding risk
FDA issues safety communication
NCCN updates anticoagulation guidance
New comparative study shows higher GI bleeding vs apixaban
Payer implements step therapy restriction
Add Field — Scenario Description (optional)

Purpose:

Clarifies mechanism and context.

Example:

FDA publicly announces formal review of post-marketing GI bleeding signals, increasing regulatory scrutiny and institutional caution.

Add Field — Expected Effect on Forecast (required)

Options:

Increases probability
Decreases probability
Mixed / uncertain

This aligns simulation logic with decision direction.

Add Field — Primary Target of Scenario

Options:

Regulators
Guideline bodies
Payers
Prescribers
Institutions
Competitors
Patients

This improves behavioral modeling.

Add Field — Evidence Basis

Options:

Regulatory communication
Peer-reviewed study
Observational data
Pharmacovigilance signal
Litigation / legal action
Guideline update
Media pressure
Internal scenario assumption

This supports confidence scoring.

Add Field — Trigger Event or Threshold

Purpose:

Defines what makes the scenario real.

Examples:

FDA formally opens safety review
New study published
Guideline update issued
Label change announced

This enables scenario validation.

Recommended Simulation Field Order

1 Scenario Name
2 Scenario Description
3 Scenario Type
4 Message Source
5 Evidence Basis
6 Primary Target
7 Expected Effect
8 Impact Level
9 Time Frame
10 Confidence Level
11 Trigger Event

2) Signals — Deduplication and Direction Logic Issues
Problem

The system allowed redundant signals and incorrect direction assignments.

Examples observed:

Duplicate signals:

Increase in GI bleeding adverse event reports in FAERS
Increase in GI bleeding adverse event reports in pharmacovigilance database

Same signal.

Should not both exist.

Required Fix — Signal Deduplication

System should automatically detect:

semantic duplicates
near-duplicate phrasing
same signal from different wording

Rule:

One signal
One object

Required Fix — Direction Validation

Example error:

FDA initiates safety review
was marked:

Supports label change

This is not always correct.

Correct logic:

Safety review initiation increases probability of regulatory action
but direction depends on outcome.

Add Signal Direction Options

Instead of binary:

Supports label change
Slows label change
Neutral

Use:

Increases probability
Decreases probability
Signals uncertainty
Signals risk escalation
Operational readiness
Market response
