Build Task — Add Agent Phase 1: National Adopter Discovery Agent

Objective

Add a bounded agent that helps answer targeting questions such as:
- Who are the most likely adopters?
- Which doctors should we target in the USA?
- Which institutions are most adoption-ready nationally?

Default scope must be the entire United States unless the user explicitly adds a geography filter.

This agent must NOT update forecast probabilities directly.
This agent must NOT write signals into active cases automatically.
It should only discover candidates, collect structured evidence, and prepare a reviewable shortlist for human approval.

Core role

The agent acts as:
- candidate discovery layer
- signal gathering layer
- structured handoff layer

It is not the forecasting engine.

Architecture placement

Question
→ National Adopter Discovery Agent
→ Candidate + signal prep output
→ Human review / validation
→ CIOS scoring engine

New component

NationalAdopterDiscoveryAgent

Inputs

- questionText
- therapyArea
- geography (optional; default = USA)
- timeHorizon
- optional targetType preference (physician | institution | both)
- optional specialty
- optional subspecialty

Outputs

- parsedQuestion
- candidateList
- candidateSignals
- candidateShortlist
- missingDataFlags
- reviewQueue

Behavior rules

1) Parse the user question into structured fields
Extract:
- therapyArea
- geography
- targetType
- specialty
- subspecialty
- timeHorizon
- adoption outcome

If geography is missing:
- set geography = USA

2) Build candidate universe nationally
Find candidate physicians and/or institutions relevant to the question across the U.S.
Initial discovery sources can be:
- hospital/service line pages
- clinical trial site listings
- conference faculty pages
- publication/investigator pages
- public program announcements
- provider directory pages

Do not require all sources in v1.
Build the architecture so source connectors can be added later.

3) For each candidate, collect structured evidence
Example evidence types:
- specialty
- subspecialty
- institution
- geography
- procedural capability
- structural heart / EP / service-line affiliation
- trial participation
- institutional readiness
- public innovation signals
- referral / workflow clues

4) Normalize evidence into candidate signals
Each signal should include:
- signalId
- candidateId
- signalType
- direction (positive | negative | neutral)
- strength (low | medium | high)
- reliability (low | medium | high)
- signalScope (institution | physician | subspecialty | specialty | market)
- sourceLabel
- sourceUrl
- evidenceSnippet
- observedAt
- eventFamilyId (optional)
- status = "candidate"

5) Produce a reviewable shortlist
For each candidate, show:
- candidate name
- candidate type (physician | institution)
- specialty
- subspecialty
- institution
- geography
- positive signals count
- negative signals count
- evidence completeness
- suggested next action

Suggested next actions:
- send to CIOS scoring
- needs review
- insufficient evidence

6) Human validation gate
No candidate signal should enter active forecasting automatically.
All discovered signals must remain in:
status = "candidate"
until a user validates them.

7) No autonomous forecast updates
The agent must not:
- compute final posterior
- update current cases automatically
- assign final LR values automatically
- modify recommendation output automatically

Initial shortlist ordering rules

Do NOT use posterior math.
Use a simple prep-score to rank discovery results for review.

Example prep-score factors:
- relevant specialty/subspecialty match = positive
- procedural capability present = positive
- institution readiness present = positive
- trial/investigator/public innovation signal = positive
- incomplete profile = penalty

This prep-score is only for ordering the review list.
It is not the final forecast.

Safety / trust rules

- Every signal must show sourceLabel and sourceUrl
- Every evidence snippet must be inspectable
- Every candidate signal must remain reviewable before use
- No hidden autonomous updates
- No silent insertion into active forecasts

Deliverables

- CandidateEntity model

- candidate discovery service

- review queue UI

Important non-goals for v1

Do NOT build:
- autonomous forecast updating
- autonomous LR assignment
- autonomous signal deduping across all cases
- full external crawling infrastructure
- automatic physician-level final ranking inside CIOS

v1 goal

Create a bounded national discovery agent that reduces manual research time for identifying likely U.S. adopters and prepares structured, reviewable candidate evidence for CIOS.
