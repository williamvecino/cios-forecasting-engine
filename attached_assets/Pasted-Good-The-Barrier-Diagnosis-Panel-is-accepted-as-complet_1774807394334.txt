Good. The Barrier Diagnosis Panel is accepted as complete.

## Decision

Proceed to the next canonical build step: **Readiness Timeline**.

## Purpose

Move from “who will adopt” and “what is blocking them” to **when adoption is likely to move, what conditions must change first, and what sequence of readiness milestones should be expected**.

This panel should translate the forecast, segmentation, and barrier logic into a time-ordered readiness view.

## What this panel must answer

At minimum:

* How ready is the market now?
* What must happen before adoption can materially accelerate?
* Which milestones are gating readiness?
* Which segments become ready first versus later?
* What is likely to move in the next 3, 6, 12, and 18 months?
* What is the most likely adoption sequence if current conditions persist?

## Build requirements

### 1) Define readiness schema

Create a readiness object that can support at minimum:

* `ReadinessID`
* `CaseID`
* `SegmentID` (nullable for overall timeline milestones)
* `MilestoneName`
* `MilestoneCategory`
* `ExpectedTimeWindow`
* `CurrentStatus`
* `ReadinessScore`
* `GatingBarriers`
* `RequiredSignals`
* `SupportingSignals`
* `CounterSignals`
* `EstimatedImpactOnAdoption`
* `ConfidenceLevel`
* `PriorityRank`
* `RationaleSummary`

### 2) Initial milestone categories

Support a practical first-pass readiness taxonomy grounded in pharma adoption logic.

Examples:

* Evidence Consolidation
* Guideline Movement
* Access / Reimbursement Change
* Operational / Workflow Readiness
* KOL / Peer Diffusion
* Community Translation
* Competitive Displacement
* Supply / Manufacturing Readiness
* Sales / Field Readiness
* Account Pathway Enablement

### 3) Overall + segment-level timeline

The panel should support:

* an overall readiness timeline for the case
* segment-specific readiness views

I want to see whether Academic KOLs become “ready” before Community Cautious Adopters, and what is gating each path.

### 4) Output required for each milestone

For each readiness milestone show:

* milestone category
* expected time window
* readiness score
* what is still gating it
* what evidence supports the timing
* what could accelerate it
* what could delay it

### 5) Time windows

Use a practical first-pass horizon structure such as:

* Now / Immediate
* 0–3 months
* 3–6 months
* 6–12 months
* 12–18 months
* 18+ months

Do not overcomplicate the temporal model yet. Functional sequencing first.

### 6) Relationship to prior layers

The Readiness Timeline must be derived from:

* forecast outputs
* adoption segments
* barrier diagnosis
* dependency-controlled signals
* reference patterns where relevant

Do not make it a generic project timeline or task tracker.

### 7) Priority / sequence logic

The system should distinguish:

* milestones already substantially ready
* milestones that are blocked but unlockable
* milestones dependent on prior events
* milestones unlikely to move within the forecast horizon

The panel should help answer:
**What has to happen first before broader adoption can occur?**

### 8) UI requirements

Need a usable panel that shows:

* overall readiness timeline
* segment-level timeline view
* milestone cards or rows grouped by time window
* gating barriers per milestone
* drilldown with rationale and supporting/counter-signals

Functional first, polished later.

### 9) Guardrail

Do not let this become a generic Gantt chart or project-management screen.

The Readiness Timeline must remain tied to:

* forecast logic
* barrier logic
* segment readiness
* explicit signals and gates

## Deliverables

Return with:

1. readiness schema
2. milestone taxonomy
3. overall readiness timeline working
4. segment-level readiness timeline working
5. sequencing / gating logic working
6. usable panel UI
7. clear note on how timeline outputs are derived from forecast + segment + barrier logic
