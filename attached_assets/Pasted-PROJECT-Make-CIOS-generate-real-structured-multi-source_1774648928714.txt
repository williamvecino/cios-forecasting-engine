PROJECT: Make CIOS generate real, structured, multi-source signals before forecasting

OBJECTIVE

Fix the signal-ingestion architecture so CIOS does not rely on generic placeholder signals and does not run weak forecasts from empty or incomplete signal sets.

The system must generate signals from multiple independent signal families, starting with the latest verified brand developments, then payer, competitor, patient, provider/behavioral, and operational sources.

This must work for any named brand or asset.
Archetype logic should guide interpretation, not replace real brand-specific evidence.

CORE PRINCIPLE

For any named brand case:

real current brand state first
then multi-source signal ingestion
then forecast
then decision output

Do not generate generic signals before attempting real brand/context retrieval.

==================================================
1. SIGNAL ARCHITECTURE TO IMPLEMENT
==================================================

Create 6 core signal families.

Every case should ingest candidate signals from these families:

1. Brand / Clinical / Regulatory
2. Payer / Access
3. Competitor
4. Patient / Demand
5. Provider / Behavioral
6. System / Operational

Each generated signal must belong to exactly one signal family.

Add this field:
- signal_family

Allowed values:
- brand_clinical_regulatory
- payer_access
- competitor
- patient_demand
- provider_behavioral
- system_operational

==================================================
2. BRAND DEVELOPMENT CHECK — MANDATORY FIRST STEP
==================================================

For any question containing a named brand, asset, or therapy, run a Brand Development Check before generating other signals.

SOURCE PRIORITY

For named brands/assets, search in this order:

1. Official company investor / press release site
2. Official brand website
3. ClinicalTrials.gov
4. Official congress / company presentation pages
5. Other high-quality public sources only if needed

RULES

1. If the brand is recognized, attempt external lookup first.
2. Latest dated, verified developments must be converted into structured signals first.
3. Generic signals may be added only after observed brand developments are captured.
4. If no verified development is found, say so explicitly.
5. Do not fabricate specifics.

Create a top Step 2 section:
"Latest Verified Brand Developments"

This should show 1–5 most recent verified items before other signals.

==================================================
3. SIGNAL CLASSES
==================================================

Add a field:
- signal_class

Allowed values:
- observed
- derived
- uncertainty

Definitions:

observed
= directly sourced, dated, externally verified event or fact

derived
= inference from observed developments

uncertainty
= unresolved issue to monitor

Observed signals should carry more weight than derived signals.
Derived signals should carry more weight than uncertainties.

==================================================
4. SIGNAL DATA MODEL
==================================================

Every signal must include these fields:

- signal_id
- title
- signal_family
- signal_class
- source_url
- source_type
- observed_date

- direction (positive/negative/neutral)
- impact (low/medium/high/very_high)

- implication

source_type allowed examples:
- official_company
- official_brand_site
- clinicaltrials
- guideline
- payer_policy
- scientific_publication
- conference
- inferred

7. SIGNAL COVERAGE CHECK — MANDATORY
==================================================

Before running a forecast, run a Signal Coverage Check.

RULE

The forecast can still render a prior probability if needed, but the system must explicitly check and display signal-family coverage.

Minimum desired coverage:
- at least 1 signal from each family

If any families are missing, show:
"Missing signal families: ..."
and list them.

Example:
Missing signal families:
- payer_access
- provider_behavioral
- system_operational

This must appear clearly in Step 2 and Step 3.

==================================================
8. DRIVER MAPPING RULES
==================================================

Signals do not go directly to forecast math.
They must map into structured drivers first.

Each driver must include:
- driver_id
- driver_name
- related_signal_ids
- direction
- impact
- reliability
- is_primary_driver (true/false)
- validation_status

Primary drivers should only be created when:
- at least one structured signal supports them
- direction exists
- impact exists
- reliability exists

No forecast sensitivity analysis unless at least one validated primary driver exists.

==================================================

11. QUESTION-TYPE AWARE SIGNAL GENERATION
==================================================

The signal generator must adapt to question type.

If question type = comparative
then generate comparative signals:
- group differences
- specialty differences
- workflow differences
- access differences
- behavior differences

If question type = ranking
then generate leader/laggard signals:
- what makes one group lead
- what delays others

If question type = threshold
then generate threshold-relevant signals

If question type = timing
then generate timing and trigger-event signals

Do not use the same signal template for all question types.

==================================================
12. CASE-TYPE / THERAPEUTIC-AREA GUARDRAILS
==================================================

Archetype cases are examples for reasoning patterns, not generic rules.

Implement this rule:
Archetype logic informs weighting and interpretation, but real brand/context signals always take precedence.

Examples:
- oncology archetype should prioritize evidence/guideline/access patterns
- dermatology adjunct archetype should prioritize visible results/safety/workflow/patient demand
- CNS LAI archetype should prioritize infrastructure/adherence/coverage
- ID hospital archetype should prioritize institutional approval/stewardship/workflow

But no archetype should overwrite actual brand developments or live signals.
