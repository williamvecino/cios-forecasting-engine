FEATURE 1 — Explain Box

Purpose: Let the user ask simple questions about any output and get a traceable answer grounded in the current case.

Think:

“What does this mean?”
“Why did the system say this?”
“What would change this?”

Where to place it

Top-right of the Judge page.

Persistent.

Always visible.

Label:

Explain
or
Ask the System

User interaction

User types:

What does operational readiness mean?

System responds:

Operational readiness refers to execution conditions required to deliver the therapy smoothly.

Primary drivers currently affecting readiness:

Prior authorization predictability
Workflow integration
Staff training completion

Improving any of these would likely increase the adoption outlook.

Behavior rules

The Explain Box must:

Use current case state
Pull from audit trail
Reference drivers and analogues
Never invent new logic
Never change the forecast
Minimal implementation steps
Step 1 — Add UI component

Create:

ExplainBox.tsx

Place on:

Judge page header

Step 2 — Add input and response panel

Required fields:

ExplainQuery {

  question: string

  caseId: string

}
Step 3 — Route query to explanation service

Create:

explain-service.ts

Logic:

function explain(question, caseState) {

  if (question contains "what does") {
     return definitionLookup(term)
  }

  if (question contains "why") {
     return reasoningFromAuditTrail(caseState)
  }

  if (question contains "what would change") {
     return constraintDriverResponse(caseState)
  }

}
Step 4 — Connect to existing data

Explain responses must pull from:

constraint drivers
forecast reasoning
analogues
audit trail

Not from external AI calls.

Display format

Show:

Definition
Current relevance
Drivers
What would change outcome

Example response

Term:

Operational readiness

Meaning:

Ability of sites to deliver the therapy smoothly.

Current limiting drivers:

Prior authorization variability
Workflow integration complexity

What would change the outlook:

Standardized authorization criteria would likely increase adoption probability.

Enforcement rule

If the system cannot answer confidently:

Return:

Insufficient information to explain this term.

Do not guess.

That is graceful failure.

FEATURE 2 — Clickable Analogues

Purpose: Show real precedent cases used to inform judgment.

This is a decision-support tool.

Not a definition tool.

Where analogues appear

Anywhere an analogue is referenced:

Analog 1
Comparable Case
Reference Case

These labels must be clickable.

What happens when clicked

A small modal window opens.

Shows the structured precedent case.

Mandatory analogue fields

Every analogue must include:

Brand
Company
Indication
Similarity
Outcome
Lesson
Confidence

If any is missing:

Do not display the analogue.

Minimal schema
AnalogCase {

  brand: string

  company: string

  indication: string

  similarity: string

  outcome: string

  lesson: string

  confidence: "High" | "Moderate" | "Low"

}
UI behavior

User clicks:

Entresto — Novartis

Window opens.
