The real fix

You need to stop treating this as “tune the logic again.”

The next phase has to be:

judgment integrity enforcement

Meaning:

1) One authoritative judgment object

Create one final structured object that contains only:

active signals
weighted signal result
gate states
raw forecast
post-gate forecast
confidence inputs
final recommendation
rationale spans

Everything on the Judge page must render from that one object.

Not from separate calculations.

2) No free-text narrative generation

Narrative must be assembled from structured fields.

For example:

if positives > negatives and no hard gate exists, it cannot say “barriers outweigh support”
if uncertainty is high, confidence cannot display “high”
if guideline support is strong, text cannot say it is weak
3) Runtime audit block

Each judgment run should expose:

signal count by direction
active gate caps
pre-gate score
post-gate score
top three positive drivers
top three negative drivers
confidence drivers
recommendation rule triggered

Until this is visible, you will keep chasing ghosts.

4) Integrity tests, not just feature tests

You now need a locked test suite that checks for contradictions like:

strong gate + positive majority + high brand strength -> cannot produce “low adoption most probable” unless a hard cap exists
uncertainty language -> cannot pair with high confidence
updated Step 2 signal state -> must exactly match Step 3 calculation inputs
5) Dependency map

You need an explicit map of:

what updates what
what is derived from what
what can be cached
what must be recomputed

Right now that dependency chain is almost certainly too loose.
