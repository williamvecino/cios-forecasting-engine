CIOS vNext Build Task — Add Target Resolution Layer Without Changing Core Forecast Math

Objective
Add hierarchical targeting to CIOS so forecasts can run at:
- market
- specialty
- subspecialty
- institution
- physician

Critical rule
Do NOT change the Bayesian/posterior math, LR logic, calibration logic, or scenario engine core.
This is a schema + filtering + UI extension only.

What to build

1) Add target hierarchy fields to the Case model
Add:
- targetType: "market" | "specialty" | "subspecialty" | "institution" | "physician"
- targetId: string | null
- specialty: string | null
- subspecialty: string | null
- institutionName: string | null
- geography: string | null

2) Create a new TargetEntity model/table
Fields:
- id
- targetType
- targetName
- specialty
- subspecialty
- institutionId
- institutionName
- physicianName
- physicianNpi (optional)
- geography
- segmentTags (array)
- isActive

3) Add scope fields to each signal
Add:
- signalScope: "market" | "specialty" | "subspecialty" | "institution" | "physician"
- appliesToTargetId: string | null
- appliesToSpecialty: string | null
- appliesToSubspecialty: string | null
- appliesToInstitutionId: string | null
- appliesToGeography: string | null
- eventFamilyId: string | null

4) Add signal eligibility filtering before forecast computation
Rule:
- market case can use market-level signals by default
- specialty case can use market + specialty signals
- subspecialty case can use market + specialty + subspecialty signals
- institution case can use market + specialty + subspecialty + institution signals
- physician case can use market + specialty + subspecialty + institution + physician signals

Do not allow physician-only signals to roll up automatically into market-level cases.

5) Add correlation guardrail
If multiple signals share the same eventFamilyId, do not stack them blindly.
Use one of these approaches:
- keep highest-weight signal only
or
- cap combined effect for same family

6) Update Case Create/Edit UI
Add these fields:
- target type
- specialty
- subspecialty
- institution
- physician
- geography

7) Update Signal Entry UI
Allow each signal to be tagged with:
- scope
- target linkage
- event family

8) Update Case Detail Header
Display target context above the forecast card:
- target type
- specialty
- subspecialty
- institution
- physician
- geography

9) Keep scenario simulator math unchanged
Only change signal availability so the simulator only shows signals eligible for the current target scope.

10) Add test cases
Create 5 sample cases for the same therapy:
- one market-level
- one specialty-level
- one subspecialty-level
- one institution-level
- one physician-level

Validate:
- forecast still computes
- signal filtering works correctly
- physician-level signals do not leak into market-level cases
- same eventFamilyId does not over-stack

Deliverables
- updated schema/models
- migration
- updated case form
- updated signal form
- signal-scope filter function
- test seed data
- no change to posterior math
