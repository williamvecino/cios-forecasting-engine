Good. The 7-step workflow and comparison-groups layer are in place.

But per canonical build order, do **not** drift into more workflow/UI expansion right now.

## Decision

The next required build remains the **Forecast Ledger**.

The new comparison-groups step is acceptable as workflow support, but it does not replace the next canonical system milestone.

## Why

We now have:

* dependency control
* compressed vs naive logic
* confidence ceiling
* comparison group definition
* improved signal framing

That means the engine is ready to start generating **tracked forecast records**.

Without the Forecast Ledger, CIOS still lacks:

* persistent forecast memory
* versioned forecast history
* resolution tracking
* calibration scoring
* institutional learning

That is the next real system value layer.

## Next required work

Build the Forecast Ledger exactly as the next milestone.

### Required functions

1. Save each forecast as a persistent ledger entry
2. Preserve forecast versions over time rather than overwriting
3. Capture the reasoning snapshot that produced the forecast
4. Allow later resolution of outcome
5. Compute Brier score when resolved
6. Support basic ledger UI and detail drilldown

### Each saved forecast must include at minimum

* ForecastID
* CaseID
* QuestionText
* ComparisonGroups
* ForecastDate
* ForecastHorizon
* PriorProbability
* PosteriorProbability
* ConfidenceLevel
* ConfidenceCeilingApplied
* EvidenceDiversityScore
* PosteriorFragilityScore
* ConcentrationPenalty
* IndependentEvidenceFamilyCount
* RawSignalCount
* CompressedSignalCount
* TopDrivers
* TopCounterSignals
* MajorLineageClusters
* EnvironmentAdjustments
* UpdateVersion
* UpdateRationale
* ResolutionStatus
* ResolutionDate
* ResolvedOutcome
* BrierScore
* CalibrationBucket

### Important requirement

The ledger must preserve the **dependency-controlled reasoning state** at the time of the forecast, not just the final probability.

That means the saved snapshot must include:

* diversity metrics
* fragility metrics
* concentration state
* confidence ceiling state
* main lineage clusters
* key positive and negative drivers

### Versioning rule

Do not overwrite prior forecasts.
Each update becomes a new version entry with timestamp and rationale.

### Resolution workflow

Support:

* Open
* Resolved True
* Resolved False
* Partially Resolved
* Not Resolvable

### Immediate deliverables

Return with:

1. Forecast Ledger schema
2. Save / load flow
3. Versioned forecast history
4. Resolution workflow
5. Brier scoring
6. Basic Forecast Ledger UI
7. Proof that dependency metrics are captured in each forecast snapshot
