Good. The Forecast Ledger core is now verified and accepted as stable.

## Decision

Proceed to the next phase: **Calibration Reference Case Library**.

## Purpose

Create a reference library of benchmark cases that helps CIOS interpret forecast performance, detect recurring structural bias patterns, and improve calibration judgment over time.

These reference cases are **not training data** and must **not directly modify the posterior engine**. They are a calibration and validation layer.

## What the Reference Case Library should do

It should provide:

* calibration anchors
* diagnostic reference cases
* structural comparison patterns
* recurring bias lessons
* benchmark scenarios for validation and interpretation

## Architectural rule

Reference cases help us answer:
**“Have we seen a similar structural pattern before, and what lessons did it teach?”**

They must **not** cause the system to mechanically copy prior outcomes or shortcut live inference.

## Build requirements

### 1) Create Reference Case schema

Each reference case should include at minimum:

* `ReferenceCaseID`
* `CaseName`
* `DecisionDomain`
* `QuestionText`
* `ComparisonGroups`
* `ForecastHorizon`
* `InitialForecast`
* `FinalForecast`
* `ConfidenceLevel`
* `EvidenceDiversityScore`
* `PosteriorFragilityScore`
* `ConcentrationPenalty`
* `IndependentEvidenceFamilyCount`
* `KeyDrivers`
* `KeyConstraints`
* `MajorLineageClusters`
* `Outcome`
* `ResolutionType`
* `BrierScore`
* `CalibrationLesson`
* `BiasPattern`
* `StructuralTags`
* `CaseSummary`

### 2) Add structured tags for cross-case comparison

Support tags such as:

* strong clinical / weak access
* workflow friction
* payer resistance
* KOL enthusiasm inflation
* false diversity
* early overconfidence
* operational constraint
* specialty ownership mismatch
* competitive disruption
* guideline acceleration
* supply / manufacturing constraint

These tags should allow pattern retrieval later.

### 3) Create first benchmark case set

Start with a small, high-quality set of manually curated archetypes.

Initial target:

* 5 to 10 reference cases

Case types should include:

* canonical success case
* canonical failure case
* false confidence case
* access-constrained case
* workflow-friction case
* competitive-disruption case
* guideline-acceleration case
* operational / supply-constraint case

### 4) Add structural lesson field

The most important field is the **CalibrationLesson**.

Lessons must be structural, not narrative.

Good example:
“Clinical strength alone did not overcome workflow friction in community settings.”

Bad example:
“The product had good efficacy.”

### 5) Add Reference Case Library UI

Need a simple library view showing:

* case name
* outcome
* structural tags
* Brier score
* key lesson

And on click:

* full case detail
* major drivers
* key constraints
* dependency pattern
* calibration lesson
* bias pattern

### 6) Add non-invasive linkage to Forecast Ledger

For now, do not let reference cases alter forecasts automatically.

But support future comparison by allowing a forecast to display:

* similar reference cases
* shared structural tags
* lessons from comparable cases

This should be informational only in this phase.

## Important restriction

Do not use the Reference Case Library to auto-adjust posterior probabilities yet.

It is a comparison and calibration aid, not an inference override.

## Deliverables

Return with:

1. Reference Case schema
2. Initial curated case set
3. Tagging system
4. Library UI
5. Example linkage from a forecast to similar reference cases
6. Clear confirmation that reference cases do not directly alter posterior calculations
